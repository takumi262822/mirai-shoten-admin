# コーディング規約

> プロジェクト: **mirai-shoten-admin（未来商店 管理画面）**  
> 作成者: Takumi Harada  
> 作成日: 2026/04/06  

---

## 1. 概要

本ドキュメントは mirai-shoten-admin における JavaScript のコーディング規約を定めます。  
全ファイルはこの規約に沿って実装・レビューすることを原則とします。

---

## 2. フォルダ構成

```
src/
├── main.js                         # エントリーポイント・UIイベント接続
├── constants/
│   ├── admin-constants.js          # 管理画面全体の定数（ストレージキー・ステータス等）
│   └── admin-constants.test.js     # 定数のユニットテスト
├── core/
│   ├── admin-manager.js            # 注文データ CRUD・統計処理（シングルトン）
│   └── admin-manager.test.js       # AdminManager のユニットテスト
├── styles/
│   └── admin-style.js              # 動的スタイル適用（カーソルグロー等）
├── ui/
│   ├── admin-components.js         # 注文テーブル・フォームの DOM 生成・更新
│   └── footer.js                   # フッター HTML 生成
└── utils/
    ├── order-validator.js          # 注文データのバリデーション関数
    ├── order-validator.test.js     # OrderValidator のユニットテスト
    ├── xss.js                      # XSS サニタイズ・安全な入力処理
    └── xss.test.js                 # XSSProtectionAdmin のユニットテスト
```

### 各ディレクトリの責務

| ディレクトリ | 責務 |
|---|---|
| `constants/` | 変更しない定数・コード定義のみを置く。ロジックを含めない |
| `core/` | ビジネスロジックの核。localStorage の CRUD を統括する |
| `styles/` | 動的スタイル操作のみ。レイアウトロジックを含めない |
| `ui/` | DOM 生成・更新のみ。ビジネスロジックを含めない |
| `utils/` | バリデーション・セキュリティなどの純粋関数 |

## 3. 命名規則

### 3-1. ファイル名

- **kebab-case** を使用する
- テストファイルは `<対象ファイル名>.test.js` とする

```
admin-manager.js       ✅
admin-manager.test.js  ✅（テスト）
adminManager.js        ❌
```

### 3-2. クラス名

- **PascalCase** を使用する

```js
export class AdminManager { }        ✅
export class OrderValidator { }      ✅
export class XSSProtectionAdmin { }  ✅
```

### 3-3. 関数・メソッド・変数名

- **camelCase** を使用する

```js
addOrder(orderData) { }        ✅
normalizeOrderRecord(r) { }    ✅
add_order(orderData) { }       ❌
```

### 3-4. 定数名

- **UPPER_SNAKE_CASE** を使用する
- アプリ全体の定数は `constants/admin-constants.js` に集約する

```js
export const AdminConstants = {
  STORAGE_KEYS: { ADMIN_ORDERS: 'admin_orders' }
};
```

## 5. コメント規約

### 5-1. ファイルヘッダー

全ファイルの先頭に JSDoc ブロックコメントを記述する。

```js
/**
 * <ファイルの役割を一行で>
 * @author Takumi Harada
 * @date YYYY/MM/DD
 */
```

### 5-2. メソッドコメント

JSDoc 形式でメソッドの役割・引数・戻り値を記述する。

```js
/**
 * 注文を追加し localStorage に保存する。
 * @param {Object} orderData - 追加する注文データ
 * @returns {{ success: boolean, data?: Object, error?: string }}
 */
addOrder(orderData) {
  ...
}
```

### 5-3. if 文コメント

**すべての `if` 文の直前行**に、何を判定しているかを日本語で説明するコメントを付ける。

```js
// インスタンスが未生成の場合にのみ新規作成する（シングルトン保証）
if (!AdminManager._instance) {
  AdminManager._instance = new AdminManager();
}
```

### 5-4. 例外処理コメント

`catch` ブロックには、なぜフォールバックするかを説明する。

```js
} catch (e) {
  // JSON.parse 失敗時は空配列にフォールバック。データ破損より起動不能の方が困る
  console.warn(`[AdminManager] LocalStorage 読込失敗: ${e.message}`);
  this.orders = [];
}
```

## 7. クラス設計

### 7-1. シングルトンパターン

`AdminManager` はシングルトンで実装する。`new` を直接呼ばず `getInstance()` 経由で使う。

```js
// ✅ 正しい取得方法
const manager = AdminManager.getInstance();

// ❌ 禁止（複数インスタンスが生まれる）
const manager = new AdminManager();
```

### 7-2. レスポンスオブジェクト

CRUD メソッドの戻り値は統一フォーマットで返す。

```js
// 成功
return { success: true, data: order };

// 失敗
return { success: false, error: 'エラーメッセージ' };
```

## 8. セキュリティ

### 8-1. XSS 対策

ユーザー入力を DOM に反映する際は必ず `XSSProtectionAdmin` のメソッドを通す。

```js
import { XSSProtectionAdmin } from '../utils/xss.js';

// ✅ 正しい使用
const safe = XSSProtectionAdmin.sanitizeInput(userInput);

// ❌ 禁止
element.innerHTML = userInput;
```

### 8-2. 入力の正規化

全角英数字・記号は `normalizeFullWidthAscii()` で半角に正規化してから保存する。

### 8-3. localStorage の入出力

- 書き込み前にバリデーションを通す
- 読み込み後は `try/catch` で包み、失敗時は空配列にフォールバックする

## 9. バリデーション

### 9-1. バリデーション関数の場所

注文データのバリデーションは `src/utils/order-validator.js` に集約する。

### 9-2. 早期リターンパターン

検証失敗時は即座に失敗レスポンスを返す。

```js
// 入力値が不正な場合は失敗レスポンスを返す
if (!OrderValidator.isValidOrder(orderData)) {
  return { success: false, error: 'バリデーションエラー' };
}
```
